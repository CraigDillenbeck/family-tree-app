SECTION 01
Rule Hierarchy
Not all design rules carry equal weight. This section classifies every foundational rule into one of three tiers so that designers and engineers can make informed decisions under time pressure, platform constraints, or edge cases — without silently eroding the system.

Tier classification
NON-NEGOTIABLE
Core to brand identity or emotional integrity. Cannot be overridden without a system-level decision and a new version of this document. Violations here are regressions, not exceptions.
DEFAULT
The correct choice in the vast majority of contexts. May be overridden in specific, documented situations where a clear rationale exists. Exceptions must be logged.
EXCEPTION-OK
A strong preference with known, acceptable edge cases. The exception path is defined here. Do not invent new exceptions — use only those listed.

SECTION 05
Practical Implementation Notes
This section pressure-tests the rules from the foundations document. Each potentially fragile rule is examined for real-world feasibility, then classified as truly non-negotiable or given a carefully controlled exception path.
0.5px borders
The rule
All UI borders are 0.5px. 1px borders are too heavy in this palette and read as interface rather than craft.
The pressure point
0.5px borders render inconsistently across devices. On non-retina (1x pixel density) screens, 0.5px rounds up to 1px or renders at ~50% opacity — which can appear invisible or inconsistent. On retina (2x+) screens, 0.5px renders correctly as a hairline.
Resolution
Non-negotiable on retina displays (target majority). On 1x displays, accept the 1px rounding — the border reads slightly heavier but remains functional. Do not detect screen density and serve different values; the complexity is not worth it. The correct implementation is: border: 0.5px solid var(--color-border-default). No media query override needed.
Controlled exceptions
Two exceptions are permitted and must use the --border-featured or --border-focus-ring tokens:
Focus rings: 2px solid rgba(28,26,23,0.20) — accessibility requirement overrides aesthetic preference.
Featured card accent (e.g. 'Recommended plan' on pricing pages): 1px solid Gold. One side only.

No-shadow policy
The rule
No drop shadows anywhere in the UI layer. Depth is expressed through tonal surface contrast only.
The pressure point
Floating elements — dropdown menus, tooltips, date pickers, command palettes — genuinely benefit from shadows because they need to visually separate from arbitrary content beneath them. Tonal contrast alone does not solve occlusion.
Resolution
One exception is permitted for floating/detached UI layers only. The shadow must use a warm ink value (never cool gray or black), be very subtle, and follow this exact spec:
box-shadow: 0 4px 16px rgba(28,26,23,0.10), 0 1px 4px rgba(28,26,23,0.06)
This exception applies to: dropdown menus, tooltip poppers, date picker calendars, command palettes, and context menus. It does not apply to cards, modals, panels, or any element anchored in the layout flow.
Modals use the --color-bg-inverse-muted scrim (rgba Ink at 40%) behind the modal card, which is anchored in a centered flex container. No shadow on the modal card itself.

Avenir Next maximum weight 500
The rule
Avenir Next 500 (medium) is the maximum weight used in the product UI. Weight 600+ is not permitted because it reads as heavy against the warm palette.
The pressure point
On inverse surfaces (Ink background), weight 500 can appear thin because the contrast inverts. Parchment text on Ink at weight 500 can look lighter than intended — particularly at small sizes and on some screens.
Resolution
Exception permitted on inverse surfaces only: Avenir Next 600 (semibold) may be used for primary labels and headings on Ink backgrounds (navigation bar, footer, inverse hero panels). This exception does not extend to the main product UI — only inverse surfaces. Token: --font-weight-semibold: 600; usage scope: inverse context only.

Tonal layering only for depth
The rule
Z-axis depth is expressed entirely through surface tonal value (Parchment → Surface-1 → Surface-2 → Surface-3). No elevation shadows.
The pressure point
The tonal system works well when surfaces share a background. It breaks when a floating panel must appear over a photograph or a rich media area — a Surface-1 card over a dark photograph may not read as elevated at all.
Resolution
Two controlled patterns for non-neutral background contexts:
Panel over photograph: use Surface-1 at full opacity with border-default. Accept that it may not look 'elevated' — treat it as a crop/frame instead.
Overlay over unknown content (e.g. image lightbox controls): use Ink background at 85% opacity with Parchment text. This is legible over any content.
The floating element shadow exception (defined above) also helps in these contexts.

Adobe Caslon web rendering
The consideration
Adobe Caslon Pro is a premium font requiring a valid Adobe Fonts license. It must be loaded via Adobe Fonts (Typekit) CDN, which introduces a dependency on an external service. Flash of unstyled text (FOUT) and font load failures must be handled.
Resolution
Font loading strategy
font-display: swap — text renders in Georgia fallback immediately, swaps to Caslon when loaded. Georgia is an acceptable fallback — warm, serif, readable.
CSS fallback stack
'Adobe Caslon Pro', Georgia, 'Times New Roman', serif
FOUT mitigation
Georgia at 17px / 1.75 line-height is close enough to Caslon that layout shift is minimal. Do not use font-display: block.
License check
Verify the Adobe Fonts plan covers web use and the domain is registered. This is a hard dependency — do not launch without it confirmed.
SSR / preload
Preload the Roman (regular) cut only. Italic is lazy-loaded. Bold cut is not needed (Caslon bold is not used in this system).

0.5px borders in CSS vs Figma
The consideration
Figma does not support 0.5px strokes natively — minimum is 1px. This creates a disconnect between design files and implementation.
Resolution
Figma workaround
Use 1px stroke at 50% opacity in Figma. Name the style 'Border/Default' and note in the style description that it renders as 0.5px in CSS. Never annotate '1px border' in specs.
CSS implementation
border: 0.5px solid var(--color-border-default) — correct and intentional.
Handoff note
All design specs should include a standing note: 'Figma borders shown at 1px/50% opacity. All borders implement as 0.5px in CSS.'
SECTION 06
Figma File Structure
The following structure governs the Prosapiam Figma workspace. All pages, their purpose, and naming conventions are defined here. This structure should be created before any components are built and treated as read-only convention once the system is in active use.
One source file. All design work references the system file via Figma's publish/subscribe model. Screen design files import components from the system. Do not duplicate component definitions across files.
File: Prosapiam — Design System
The canonical system file. Contains all foundations, tokens, and components. Published as a team library. All screen design files link to this file.

Page 1 — Cover
File title, version, last updated date, owner. Quick links to key pages. Not a design surface — administrative only.
Page 2 — Foundations
Color swatches (all semantic roles labeled). Typography specimens (all scale levels, both typefaces, showing correct usage). Spacing scale visualization. Border radius reference. Surface hierarchy diagram. This page is documentation, not components.
Page 3 — Tokens
Figma Variables panel (local variables mirroring the CSS token list). Color variables: all --color-* tokens as Figma Color Styles. Text styles: all --font-* combinations as Figma Text Styles. Effect styles: border-focus-ring, featured-border. No layout components on this page — it is a variable/style registry only.

Naming conventions
Consistent naming across Figma, tokens, and code is the contract that makes handoff trustworthy. These conventions are mandatory.

Components
PascalCase. Full category prefix. Examples: Button/Primary, Card/Person, Input/Text
Variants
Property=Value pairs in Figma variant panel. State=Default|Hover|Active|Focus|Disabled|Loading|Error
Color styles
Mirror CSS token names. Slash-separated: Color/Text/Primary, Color/Surface/1, Color/Accent
Text styles
Mirror CSS roles: Type/Display/XL, Type/H1, Type/Body/Story, Type/Label, Type/Caption
Spacing
In annotations only: Space/4 = 16px, Space/8 = 32px etc. (number = multiplier, not px value)
Layers
Descriptive, lowercase, hyphenated. 'profile-photo', 'card-header', 'nav-link-label'. Never 'Rectangle 47'.
Frames
Screen frames: [Breakpoint]/[Section]/[State]. Example: Desktop/Profile/Default
Auto-layout
All components use auto-layout. No fixed-position components except overlays.

Variable / token structure in Figma
Figma Variables should be organized in collections that mirror the CSS token groups. Use modes for light/dark theme switching.

Collection: Color
All --color-* tokens. Two modes: Light (default) and Dark. Every color variable has both mode values defined before the component library is published.
Collection: Typography
Font size, weight, line-height, letter-spacing tokens as Number variables. Font family as String variables. Reference these in Text Styles, not directly in components.
Collection: Spacing
All --space-* tokens as Number variables. Used in auto-layout padding and gap values across all components.
Collection: Radius
All --radius-* tokens as Number variables.
Collection: Motion
Duration and easing as String variables. Note: Figma does not support custom easing in prototypes — these are documentation tokens for handoff, not live prototype values.

Component build order
Build in this sequence. Each tier depends on the previous.

Tier 1 — Primitives
Color variables, text styles, effect styles. Then: Divider, Badge, Tag, Avatar, Icon wrapper.
Tier 2 — Form elements
Button (all variants + states), Input, Select, Textarea, Checkbox, Radio, Toggle.
Tier 3 — Layout
Card (all variants), Panel, Modal shell, Drawer shell.
Tier 4 — Navigation
Nav Bar (desktop + mobile), Tabs, Breadcrumb.
Tier 5 — Content cards
Person Card, Memory Card, Tree Node, Media Card.
Tier 6 — Patterns
Patterns page only — no component publication.
Tier 7 — Screens
Full screen compositions using all prior tiers.

Figma to frontend handoff checklist
Before any component is handed off to engineering, verify:
All layer names are descriptive and match the component naming convention.
All colors reference a Figma Color Style — no hard-coded hex values in components.
All text uses a Text Style — no manually set font values.
All spacing uses auto-layout with token-referenced gap and padding values.
All states are present as Figma variants: Default, Hover, Active, Focus, Disabled.
Border values are annotated with the CSS token name and the note about 0.5px implementation.
The component has been reviewed against the §03 state rules in this document.
The component has been reviewed against the §04 accessibility guardrails.
Dark mode variants exist for all components (or a note explains why they don't differ).
