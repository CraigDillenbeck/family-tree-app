PART I
Core UI Components
The components below are the foundational building blocks of the Prosapiam interface. Each is defined at the level of specificity needed to build it directly in Figma and implement it in React or HTML/CSS. Token references use the CSS custom property names established in Design System Spec v1.1.

01 — Button
The primary action trigger in the system. Three variants correspond to three levels of action weight. Buttons never use color to signal importance — only fill and outline distinguish them. All variants share the same geometry and type treatment.
Variants
Primary
Filled. bg: --color-ink. text: --color-parchment. Use for the single most important action on a screen. One primary button per view.
Secondary
Outlined. bg: transparent. border: --border-default (0.5px). text: --color-ink. Use for secondary actions alongside a primary.
Ghost
Text only. bg: transparent. no border. text: --color-ink. underline at offset 3px. Use for tertiary actions, cancel, skip, or back navigation.
Destructive
Filled. bg: --color-terra. text: --color-parchment. Use only for irreversible destructive actions (delete, remove). Requires confirmation pattern.

A11y: Role: button. All buttons require an accessible name. Icon-only buttons require aria-label. 02 — Input
Single-line text entry. The workhorse of form interactions. Visual character is quiet — the field should recede and let the user's content become the subject. Labels always sit above the field. Never floating, never placeholder-only.

With prefix

With suffix

Search
Prefix: search icon (16px). Suffix: clear button (appears when value present). Borderless option for inline search contexts.

03 — Textarea
Multi-line text entry. Used for story/memory content, biographical notes, and longer descriptions. The most emotionally significant input in the product — this is where people write about their grandparents. Its visual treatment must feel worthy of that.
Variants
Standard
Fixed or resizable. Default: vertically resizable, not horizontally. Min-height: 120px.
Story entry
Large variant for memory/story content. Min-height: 240px. Avenir Next label but content renders in Avenir Next 400 until saved — Caslon is a display typeface for reading, not editing.
Auto-grow
Height grows with content. No manual resize handle. Used in inline edit contexts.

08 — Badge
A compact semantic label. Not interactive. Communicates a categorical state, relationship type, or status. In Prosapiam, badges carry real emotional weight — 'Living', 'Deceased', 'Maternal', 'Paternal' are not just labels.

Sage
bg: Sage-Tint (#E4EDE5) · text: Sage (#5A6B5C) · Use for living status, active states
Terra
bg: Terra-Tint (#F2E4E1) · text: Terra (#8C4A38) · Use for deceased status only — handle with care
Warm
bg: Surface-2 · text: Warm-Mid · Use for metadata categories (Paternal line, Maternal line)
Geometry & typography
Font
Avenir Next 500 · 10px · ALL CAPS · 0.06em tracking · sentence case for proper nouns
Padding
3px vertical · 8px horizontal
Radius
--radius-xs (2px)
Max width
None — text should not truncate · use concise labels only
Icon
Optional 10px icon at left · 4px gap to text
Stacking

Tone: Badge labels should be chosen with care. 'Deceased' is weighted — avoid 'Dead'. 'Living' is warm — avoid 'Active'. The vocabulary of the badge system is part of the product's emotional tone.
A11y: Badges are presentational. Screen reader should read badge text inline with surrounding content — no special ARIA role needed. For status badges (Living/Deceased): include in the accessible name of the parent component so it is not read in isolation.

11 — Divider
A structural separator. Used to organize content within cards and panels. The divider is not decoration — it marks a genuine conceptual boundary. Do not use dividers between every element; use whitespace first.
Variants
Horizontal
Full-width · 0.5px · color: --color-border-default (#D6CFC4) · margin: --space-6 (24px) vertical
Horizontal muted
0.5px · color: --color-border-subtle · for lighter separation within a single content area
Vertical
1px height of parent · same color · used only in horizontal button groups or tab bars
With label
Horizontal divider with centered text label: Avenir 400 · 11px · ALL CAPS · Warm-Mid · padded 16px horizontal from line · used for timeline year dividers
Implementation: Use <hr> for semantic dividers. role='separator' if implemented as <div>. Never use a Table row as a divider. Never use margin alone to imply separation — dividers are structural, not spatial.
A11y: Decorative dividers: aria-hidden='true'. Semantic dividers (section separators in a document-like context): native <hr> is sufficient. Vertical dividers in button groups: aria-hidden='true'.

16 — Tooltip
Reveals supplementary text on hover or focus. Used for icon button labels, truncated text, date formatting, and brief contextual explanations. Tooltips contain text only — no interactive elements, no rich content.
Anatomy

Typography
Avenir Next 400 · 12px · Parchment · max-width 240px · wraps naturally
Arrow
Optional 6px caret pointing to trigger · matches bg color
Shadow
Floating element exception applies: 0 4px 16px rgba(28,26,23,0.10) + 0 1px 4px rgba(28,26,23,0.06)
Position
Top preferred · auto-flips to bottom/left/right if viewport edge approached · 8px offset from trigger
Behavior
Trigger
Hover (200ms delay before show) · Focus (immediate show) · Touch (tap-to-toggle on mobile)
Dismiss
Mouse leave · blur · Escape key · after 5s if no interaction (optional)
Animation
fade + scale(0.95→1) · 150ms --ease
Persistence

Do not use tooltip as a replacement for a visible label. Touch devices: tooltip on first tap, action on second tap for icon-only buttons — or provide visible labels on mobile.

P02 — Relationship Connector Treatment
The lines that link nodes in the family tree. Connectors are not merely decorative — they carry relationship type, direction, and sometimes status. Their visual treatment must be legible without overwhelming the nodes.
Line types
Parent–child
Solid · 1px · Warm-Light (#C4B9A8) · orthogonal routing (right-angle bends, no diagonal)
Spouse / partner
Solid · 1.5px · Gold-Light (#BFA882) · horizontal connector between two nodes at same tier
Divorced
Dashed · 1px · Warm-Light · 4px dash · 4px gap · same routing as spouse
Adopted
Dotted · 1px · Warm-Mid · 2px dot · 6px gap
Uncertain
Dashed · 0.5px · Warm-Light · lighter, more tentative appearance
Routing rules
Orthogonal only — no diagonal lines. Horizontal then vertical, or vertical then horizontal.
Junction points (where lines meet a horizontal bus line): small filled circle, 4px, Warm-Light.
Minimum clearance between parallel lines: 8px.
Lines route behind nodes, never over them — z-index of connectors below nodes.
On hover of a connector: line weight increases to 2px, color shifts to Warm-Mid. Label appears if applicable.
Connector labels
Optional label on connector: Avenir Next 400 · 10px · Warm-Mid · appears on hover only unless the relationship is non-obvious. Label sits in a small white pill (bg white, border-subtle) centered on the connector at its longest segment.
Responsive behavior
At zoom < 75%: connector labels hidden. Lines remain, reduce to 0.5px.
At zoom < 50%: only spouse connectors remain visible. Parent-child structure implied by node position.
At zoom < 25%: all connectors visible as 0.5px lines only.
A11y: Connectors are structural but not interactive. In the list-view alternative, relationships are communicated via text: 'Parent of [name]', 'Spouse of [name]'. Connectors themselves are aria-hidden='true' in the SVG layer.

P08 — Add Relationship Flow
A multi-step interaction for connecting two people in the tree. This is more complex than a simple form — it requires selecting or creating a person, defining the relationship type, and confirming the connection. The flow must be learnable on the first use and fast on the tenth.
Step structure
Step 1: Select person
Choose who to connect to the current person. Search field with autocomplete showing Person Summary Cards in results. Option: 'Add a new person' if they don't exist yet (inline mini-form or branches to Add Person modal).
Step 2: Define relationship
Select relationship type: Parent of · Child of · Spouse / Partner of · Sibling of · 'Other' (with custom text). Direction is explicit — 'Add [Name A] as a parent of [Name B]'. Confirmation sentence auto-generates.
Step 3: Confirm
Preview card showing both people and the connector type. 'Confirm' (Primary) or 'Back' (Ghost). No data is saved until this step.
UI pattern
Implemented as a stepped modal (stepper pattern within the Wide Modal component).
Step indicators: three dots at modal top — filled dot for current step, hollow for future, checkmark for completed.
Progress is preserved if user navigates back — filled fields and selections retained.
'Back' always available on steps 2 and 3.
Search / autocomplete behavior
Minimum 1 character to trigger search. Results appear below input in dropdown list.
Each result: Person Summary Card (compact horizontal — Avatar + Name + Dates).
No results: 'No one found — add [search term] as a new person?' with action link.
Recent/suggested: if no query typed yet, show 5 most recently added people.
Responsive behavior
Desktop: modal stays wide. Steps are side-by-side summary pane (right) + active step (left).
Mobile: fullscreen modal. Steps are sequential full-screen — no split layout.
A11y: Stepper: aria-label='Add relationship, step N of 3' on modal. Step indicators: aria-current='step' on active dot. Search autocomplete: combobox ARIA pattern (same as Select component). Confirmation sentence (Step 3): read aloud by screen reader as a summary before submission.
